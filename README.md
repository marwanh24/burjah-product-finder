# مساعد اختيار منتجات بُرجة العربية

ويدجت داخل صفحة سلة يسأل الأم 6 أسئلة (عمر، مناسبة، ستايل، لون بشرة، ألوان مفضّلة، ميزانية)
ثم يعرض 3 توصيات (أفضل/أبسط/أفخم) بأسعار ومقاسات وتوفر حقيقي من كتالوج متجر بُرجة العربية.

مبني حسب حزمة `START_HERE_claude_code.md` + `logic_prompt.md` + `build_spec.md` (مرفقة أصلاً بالطلب).

## البنية

```
data/           منطق ثابت: products.json (خريطة العائلات/المناسبات/الألوان) + rankings.json (ترتيب العائلات)
                 يُولَّدان من build-catalog.js (مصدر الحقيقة القابل للصيانة — أعد تشغيله عند إضافة منتج)
backend/        خدمة Node/Express على Render: تُزامن مع Salla كل 6 ساعات وتكتب live.json (سعر/مخزون/توفر حقيقي)
engine/         محرك القرار (decisionEngine.js) — شجرة ثابتة بالكامل، بلا أي نموذج ذكاء اصطناعي
widget/         الويدجت (HTML/CSS/JS مضمّن بالكامل) — يُبنى بـ widget/build.js إلى widget/dist/burjah-widget.html
```

## التشغيل والاختبار

```bash
# 1) توليد الكتالوج الثابت من قواعد logic_prompt.md
node data/build-catalog.js

# 2) اختبارات الباك-إند (فكّ الاسم، الترقيم، سرّية cost_price، اكتشاف الخصم)
cd backend && npm install && npm test

# 3) اختبارات محرك القرار (حالات وهمية: ميزانية، طقم، اختر لي، حياد لون البشرة، لا نتيجة)
node engine/test/decisionEngine.test.js

# 4) بناء الويدجت النهائي
node widget/build.js   # -> widget/dist/burjah-widget.html
```

## نشر الباك-إند (Render)

1. أنشئ خدمة Web جديدة على Render من هذا المستودع، أو استخدم `backend/render.yaml` مباشرة (Blueprint).
2. أضف متغيّر البيئة `SALLA_API_TOKEN` يدوياً من لوحة Render (لا يُكتب في أي ملف مرفوع).
3. بعد النشر، الخدمة تعرض:
   - `GET /live.json` — بيانات نظيفة (بلا cost_price، بلا أسرار)، CORS مقصور على `ALLOWED_ORIGIN` (افتراضياً burjah.com).
   - `GET /health` — فحص صحة + وقت آخر مزامنة.
   - مزامنة تلقائية عند الإقلاع ثم كل 6 ساعات.

## تسليم الويدجت لسلة

1. افتح `widget/dist/burjah-widget.html` وعدّل رابط `LIVE_JSON_URL` (أعلى الملف، داخل وسم `<script>` الأخير) إلى رابط خدمة Render الفعلي.
2. الصق محتوى الملف كاملاً في خانة "محتوى HTML" بمحرر ثيم سلة.
3. لا حاجة لأي مفتاح API أو إعداد إضافي — كل شيء مضمّن.

## اختبار حي أُجري فعلاً

تم التحقق من الأداة مقابل Salla MCP الحي فعلياً (لا افتراضات) على 3 منتجات: ثوب الهيبة (2142939960)،
ثوب العز (1434664687)، ثوب العراقة (1179255884). تأكد فعلياً:
- الترقيم يقف عند 60 عنصراً/صفحة (الهيبة: 62 متغيّراً على صفحتين) — الباك-إند يجلب كل الصفحات.
- حقل `sku` فارغ دائماً؛ المفتاح الحقيقي `variant.id` + `barcode`.
- حقل `name` قد يأتي مكرراً مشوّهاً (مثل `"49 / أبيض / 49 / 49..."`) — الفاصل يأخذ أول رقم/أول نص فقط.
- `cost_price` موجود فعلياً في استجابة Salla (على مستوى المنتج والمتغيّر) — يُستبعد بالكامل من live.json.

### تصحيح مهم عن برومت البناء الأصلي
`build_spec.md` افترض أن الخصم يُكتشف عبر حقل `has_special_price` على مستوى المنتج. الاختبار الحي
أثبت أن هذا الحقل **غير موجود إطلاقاً** على مستوى `/products/{id}` (فقط داخل كل `sku` فرعي). الخصم
الحقيقي يُكتشف بمقارنة `regular_price` (السعر قبل الخصم) بـ `price` (السعر الحالي الفعلي) — وهذا ما
ينفّذه `backend/src/sync.js` فعلياً، وتؤكده اختبارات `backend/test/run-tests.js`.

## قرارات هندسية اتُّخذت أثناء البناء (تستحق مراجعة المالك)

1. **سؤال الميزانية**: موك-أب الواجهة (`build_spec.md §5.4`) عدّد 5 شاشات أسئلة فقط ولم يذكر
   الميزانية، بينما `logic_prompt.md` يُلزم صراحة بعدم التوصية دون ميزانية. أضفتُ شاشة سادسة خفيفة
   (حقل رقم واحد) قبل النتائج مباشرة لتحقيق هذا الشرط الإلزامي دون الإخلال بروح التصميم المعتمد.
2. **دمج "عيد أو زيارة" و"مدرسة أو يومي"**: هذان زرّان في موك-أب السؤال الثاني، لكن "عيد" و"زيارة"
   شرطان صلبان منفصلان في `logic_prompt.md` (بعض العائلات مؤهلة لأحدهما دون الآخر). نفّذتُ الزر
   كمجموعة "أو" (المنتج يمر إن كان مؤهلاً لأي من الاثنين) — يطابق نيّة الزر دون كسر الشروط الصلبة.
   "مدرسة/يومي" دمجهما آمن تماماً لأنهما يظهران معاً دائماً في كل قواعد العائلات.
3. **السديري المستقل غير مفعّل حالياً في الويدجت**: يتطلب منطقياً تأكيد أن الأم تملك ثوباً مناسباً
   (`logic_prompt.md §10.1`) — وهذا سؤال لا يطرحه الموك-أب الحالي. المحرك يدعمه بالكامل (عبر
   `answers.hasThobForVest === true`)، لكنه لن يُقترح تلقائياً ضمن "اختاروا لي" حتى تُضاف هذه الخطوة
   للواجهة مستقبلاً — قرار متحفظ لتفادي افتراض غير مؤكد.
